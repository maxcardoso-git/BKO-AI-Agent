## Hotfixes aplicados nesta sessão (BKOConsole)

### 1. `/admin/tokens` retornava "Acesso não autorizado"
- **Causa:** `/admin/tokens` e `/admin/locks` não estavam em `PREFIX_RULES` em `route-roles.ts`
- **Fix:** Adicionadas as entradas com roles `ADM` e `ADM_SUP` respectivamente
- **Arquivo:** `BKOConsole/src/config/route-roles.ts`

### 2. Link de token do operador (`/processar?token=xxx`) retornava "Acesso não autorizado"
- **Causa 1:** `/processar` também estava faltando em `route-roles.ts` → `getRequiredRolesForPath` retornava `null` → `useRequireAuth(null)` redireciona para `/unauthorized`
- **Causa 2:** Middleware redirecionava `/processar` sem sessão para `/login` descartando o `?token=` da URL
- **Causa 3:** `AppLayout` bloqueava com `!hasAccess` antes da troca do token acontecer

- **Fix route-roles.ts:** Adicionado `{ prefix: '/processar', roles: OP_SUP_ADM }`
- **Fix middleware.ts:** Condição `!hasTokenParam` adicionada para deixar passar `/processar?token=xxx` sem redirecionar para login
- **Fix app-layout.tsx:** Extraído `AppLayoutInner` com `useSearchParams()`; quando `?token=` presente passa `[]` para `useRequireAuth` e não bloqueia renderização. Envolvido em `<Suspense>` (obrigatório para `useSearchParams`)

<decisions_made>

- Middleware usa `request.nextUrl.searchParams.has('token')` para detectar link de operador — só `/processar` tem esse fluxo, mas a verificação é genérica no middleware (inofensiva para outras rotas)
- `AppLayout` dividido em `AppLayout` (wrapper com Suspense) + `AppLayoutInner` (lógica) — `useSearchParams()` exige Suspense boundary no Next.js 16
- Quando `?token=` presente, passa `[]` (não `null`) para `useRequireAuth` — `[]` significa "qualquer role autenticado", enquanto `null` significa "proibido"
- `/admin/locks` recebeu `ADM_SUP` (ADMIN + SUPERVISOR) não só ADMIN, alinhado com o padrão das outras páginas operacionais de administração

</decisions_made>
