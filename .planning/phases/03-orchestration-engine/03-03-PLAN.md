  <name>Task 1: Create TicketExecutionController with BFF endpoints</name>
  <files>backend/src/modules/execucao/controllers/ticket-execution.controller.ts</files>
  <action>
Create `ticket-execution.controller.ts` in the existing controllers directory.

The controller injects `TicketExecutionService` via constructor.

All endpoints are auto-protected by the global JwtAuthGuard (APP_GUARD pattern from Phase 2). No @Public() decorator needed — operators must be authenticated.

**Endpoint 1: POST /api/complaints/:id/executions**
```typescript
@Post('complaints/:id/executions')
@HttpCode(201)
async startExecution(@Param('id') complaintId: string): Promise<TicketExecution> {
  return this.ticketExecutionService.startExecution(complaintId);
}
```

**Endpoint 2: POST /api/executions/:id/advance**
```typescript
@Post('executions/:id/advance')
async advanceStep(
  @Param('id') executionId: string,
  @Body() body?: { operatorInput?: Record<string, unknown> },
): Promise<StepExecution> {
  return this.ticketExecutionService.advanceStep(executionId, body?.operatorInput);
}
```

**Endpoint 3: POST /api/executions/:id/finalize**
```typescript
@Post('executions/:id/finalize')
async finalizeExecution(
  @Param('id') executionId: string,
  @Body() body?: { outcome?: Record<string, unknown> },
): Promise<TicketExecution> {
  return this.ticketExecutionService.finalizeExecution(executionId, body?.outcome);
}
```

**Endpoint 4: POST /api/executions/:id/retry-step**
```typescript
@Post('executions/:id/retry-step')
async retryStep(
  @Param('id') executionId: string,
  @Body() body: { stepKey: string },
): Promise<StepExecution> {
  return this.ticketExecutionService.retryStep(executionId, body.stepKey);
}
```

Use `@Controller('api')` as the base path to match existing conventions (the ExecutionController uses `@Controller('api')`). Check the existing ExecutionController for the exact pattern used — if it uses `@Controller('api/complaints')` then adjust accordingly to avoid path conflicts.

IMPORTANT: The route `complaints/:id/executions` must NOT conflict with the existing ExecutionController's GET routes. The existing controller handles `GET /api/complaints/:id/executions` — this new controller handles `POST /api/complaints/:id/executions`. Same path, different HTTP method. NestJS handles this correctly even across controllers.

Import `Controller, Post, Param, Body, HttpCode` from `@nestjs/common`.
Import `TicketExecutionService` from the services directory.
Import `TicketExecution` and `StepExecution` entity types for return type annotations.

**End-to-end verification sequence:**

First, get an auth token (test user from seed data):
```bash
TOKEN=$(curl -s -X POST http://localhost:3001/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"operator@bko.com","password":"operator123"}' | jq -r '.access_token')
```

Then, find a complaint with a tipologyId set (from seed data):
```bash
COMPLAINT_ID=$(curl -s http://localhost:3001/api/complaints \
  -H "Authorization: Bearer $TOKEN" | jq -r '.data[0].id')
```

Step 1 — Start execution:
```bash
curl -s -X POST "http://localhost:3001/api/complaints/$COMPLAINT_ID/executions" \
  -H "Authorization: Bearer $TOKEN" | jq .
```
Expect: 201, status="running", currentStepKey is set, metadata has tipologyKey + slaDeadline.

Step 2 — Advance step (repeat for each step in the capability):
```bash
EXEC_ID=$(curl -s -X POST "http://localhost:3001/api/complaints/$COMPLAINT_ID/executions" \
  -H "Authorization: Bearer $TOKEN" | jq -r '.id')
# (use the id from step 1 if already started)

curl -s -X POST "http://localhost:3001/api/executions/$EXEC_ID/advance" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" | jq .
```
Expect: 200, stepKey is set, status="completed", output has stub data.

Step 3 — Continue advancing until execution completes or hits a human-required step.

Step 4 — Finalize (if not auto-completed):
```bash
curl -s -X POST "http://localhost:3001/api/executions/$EXEC_ID/finalize" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" | jq .
```
Expect: 200, status="completed".

Step 5 — Verify concurrent guard:
```bash
curl -s -X POST "http://localhost:3001/api/complaints/$COMPLAINT_ID/executions" \
  -H "Authorization: Bearer $TOKEN" -w "\n%{http_code}"
```
If previous execution is still RUNNING, expect 409.

NOTE: If the backend does not start (dependency issues), debug and fix. Check for:
- "Nest can't resolve dependencies" — means a module import is missing
- Port conflict on 3001 — kill existing process first
- Database connection — ensure Docker postgres is running on port 5433

If a seeded complaint does not have a tipologyId, the start will fail with 422. In that case, verify seed data or use a complaint that has tipologyId set.
