<objective>
Reduce the regulatory pipeline from 16 active steps to 14 by deactivating `retrieve_discounts` and `retrieve_invoices`, and update the LoadComplaint skill to read the latest active operator note and emit `enrichedText` so downstream skills can consume the operator's contextual brief.

Purpose: PIPE-01 (deactivate 2 steps), PIPE-02 (LoadComplaint emits enrichedText), and PIPE-05 (BuildMandatoryChecklist tolerance). After this plan, every ticket execution runs the simplified 14-step flow and downstream skills (DraftFinalResponse in Plan 08-03) can read enrichedText. Also lays the foundation for AUDIT-TIMING-05 by emitting the first lifecycle event (`ticket_created`) directly inside LoadComplaint.

Output: 1 SQL UPDATE migration + 1 modified skill (loadComplaint). No module wiring changes are needed — both `ComplaintUserNote` and `TicketTimingEvent` were registered in OperacaoModule by Plan 08-01, and ExecucaoModule already imports OperacaoModule (so `@InjectRepository(...)` resolves transitively).
</objective>
