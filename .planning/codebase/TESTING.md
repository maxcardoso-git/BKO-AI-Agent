# Testing Patterns

**Analysis Date:** 2026-05-06

## Honest Summary: Coverage is Effectively Zero

This codebase has the Nest CLI test scaffolding in place but **no meaningful automated test coverage**. Only the boilerplate "hello world" tests generated by `nest new` exist. There are no unit tests for any business module, no integration tests, no frontend tests, and no E2E coverage of real flows.

**Scope of existing tests:**
- 1 unit spec: `backend/src/app.controller.spec.ts` — asserts `getHello()` returns `'Hello World!'`.
- 1 e2e spec: `backend/test/app.e2e-spec.ts` — asserts `GET /` returns `200` with body `'Hello World!'`. **This test is broken in practice** because `main.ts` sets `setGlobalPrefix('api')`, so `GET /` no longer responds; only `GET /api` would.
- 0 tests across all of `backend/src/modules/**` (auth, operacao, execucao, orquestracao, ia, memoria, regulatorio, base-de-conhecimento).
- 0 tests across all of `frontend/src/**`.

Treat any new phase that touches business logic, skills, orchestration, or UI flows as introducing logic that has **no regression safety net**. Adding tests should be considered when modifying critical paths.

## Test Framework (Backend)

**Runner:** Jest 30 (`backend/package.json` devDependencies).

**Transformer:** `ts-jest` 29.

**Config (inline in `backend/package.json` under `"jest"`):**
```json
{
  "moduleFileExtensions": ["js", "json", "ts"],
  "rootDir": "src",
  "testRegex": ".*\\.spec\\.ts$",
  "transform": { "^.+\\.(t|j)s$": "ts-jest" },
  "collectCoverageFrom": ["**/*.(t|j)s"],
  "coverageDirectory": "../coverage",
  "testEnvironment": "node"
}
```
- Unit specs must live under `backend/src/` and match `*.spec.ts`.
- E2E specs use a separate config: `backend/test/jest-e2e.json` (`rootDir: '.'`, `testRegex: '.e2e-spec.ts$'`).

**NestJS testing utilities:** `@nestjs/testing` (devDependency). HTTP assertions use `supertest` + `@types/supertest`.

**No assertion library beyond Jest** — uses built-in `expect`.

**Run commands (from `backend/`):**
```bash
npm test               # Run all unit specs (src/**/*.spec.ts)
npm run test:watch     # Jest in watch mode
npm run test:cov       # With coverage report -> ../coverage/
npm run test:debug     # Inspector-attached for debugging
npm run test:e2e       # E2E suite (test/*.e2e-spec.ts)
```

## Test Framework (Frontend)

**None configured.**

`frontend/package.json` has no test runner, no test script, no Jest/Vitest/Playwright dependencies. The only `scripts` entries are `dev`, `build`, `start`, `lint`. There are no `*.test.ts(x)` or `*.spec.ts(x)` files anywhere in `frontend/src/`.

If frontend testing is added, candidates aligned with the stack (Next.js 16, React 19) would be Vitest + React Testing Library for components and Playwright for E2E. No such choice has been made yet.

## Test File Organization

**Existing layout:**
```

  test/
    app.e2e-spec.ts               # e2e specs live separately
    jest-e2e.json
```

**Convention (implied, since only one example exists):**
- Unit specs are co-located next to the source file under test (`foo.service.ts` -> `foo.service.spec.ts`).
- E2E specs live under `backend/test/` with the `.e2e-spec.ts` suffix.

## Existing Test Patterns

**Unit spec template** (`backend/src/app.controller.spec.ts`):
```typescript
import { Test, TestingModule } from '@nestjs/testing';
import { AppController } from './app.controller';
import { AppService } from './app.service';

describe('AppController', () => {
  let appController: AppController;

  beforeEach(async () => {
    const app: TestingModule = await Test.createTestingModule({
      controllers: [AppController],
      providers: [AppService],
    }).compile();

    appController = app.get<AppController>(AppController);
  });

  describe('root', () => {
    it('should return "Hello World!"', () => {
      expect(appController.getHello()).toBe('Hello World!');
    });
  });
});
```

**E2E template** (`backend/test/app.e2e-spec.ts`):
```typescript
import { Test, TestingModule } from '@nestjs/testing';
import { INestApplication } from '@nestjs/common';
import request from 'supertest';
import { AppModule } from './../src/app.module';

describe('AppController (e2e)', () => {
  let app: INestApplication<App>;

  beforeEach(async () => {
    const moduleFixture: TestingModule = await Test.createTestingModule({
      imports: [AppModule],
    }).compile();
    app = moduleFixture.createNestApplication();
    await app.init();
  });

  it('/ (GET)', () => {
    return request(app.getHttpServer()).get('/').expect(200).expect('Hello World!');
  });
});
```

## Mocking and Fixtures

**No established patterns** beyond stock Jest. There are no shared mocks, no test fixtures, no factory helpers, no test database setup, and no `beforeAll`/`afterAll` lifecycle hooks for DB resets. Any new test author has to invent the conventions.

**Recommended approach when adding tests:**
- For services: use `Test.createTestingModule({ providers: [ServiceUnderTest, { provide: getRepositoryToken(Entity), useValue: mockRepo }] })` with hand-rolled `jest.fn()` repository mocks. The `@nestjs/typeorm` package exports `getRepositoryToken` for this.
- For controllers: provide the service as `{ provide: SomeService, useValue: { method: jest.fn() } }`.
- For E2E: spin up a real Postgres (Docker) and use `supertest` against `app.getHttpServer()`. Remember `setGlobalPrefix('api')` and `ValidationPipe` from `backend/src/main.ts` — replicate the bootstrap pipeline or import from a shared helper.

## Gaps Worth Highlighting

1. **Orchestration engine** (`backend/src/modules/execucao/services/ticket-execution.service.ts`): SLA computation, step advancement, HITL gating, auto-advance loop — all untested. This is the most complex code in the system.
2. **Skill registry** (`backend/src/modules/execucao/services/skill-registry.service.ts`): every individual skill is untested; the soft-fail error path (`return { error, ... }`) is also untested.
3. **AI agents** (`backend/src/modules/ia/services/*.agent.ts`): zod schemas for structured output have no tests verifying parsing or fallback behaviour.
4. **Auth flow** (`backend/src/modules/auth/`, `frontend/src/lib/dal.ts`, `frontend/src/lib/session.ts`): JWT issuance, JWE encryption, cookie verification untested end-to-end.
5. **Frontend pages and server actions:** zero coverage. Form actions in `frontend/src/app/**/actions.ts` have hand-rolled validation that is not exercised.
6. **TypeORM entities:** no tests assert that migrations match entity definitions.

When planning future phases, consider adding focused tests for the path being modified rather than backfilling everything at once. The framework is in place — only the discipline of writing specs is missing.
